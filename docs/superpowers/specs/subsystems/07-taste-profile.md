## 1. One-liner contract

Reads `taste/tracks.csv`, computes structured aggregations over the enriched rows, and writes two artifacts: `taste/profile.md` (a prose narrative of Destin's taste) and a fixed list of 14 `knowledge/artists/<slug>.md` cheat sheets covering the artists most-represented in the data plus the master-spec-named taste anchors. Owns both output locations and re-runs on a conversational trigger only.

## 2. Master-spec relationship

This subsystem **does not override** any master-spec decision. It honors:

- **§7.4** as the prose shape of `taste/profile.md` exactly (TL;DR, BPM clusters, Key bias, Top artists & subgenres, Resources surface area), with a YAML front-matter block prepended for traceability (§3.5).
- **§8** cadence "regenerate when tracks.csv row count changes ≥10%, or on demand." This subsystem implements only the **on-demand** path (conversational trigger from Destin). The 10%-row-count automatic check is intentionally not implemented in v1: profile regeneration is a Claude-mediated workflow that takes minutes and produces 15 markdown files; gating it on Destin's explicit ask matches the established pattern set by Subsystems #5 (Spotify pull) and #6 (audio enrichment) and avoids surprise regenerations. If `taste/profile.md` becomes stale relative to `tracks.csv`, downstream Subsystem #4 surfaces that drift via its `generated_from_profile` front-matter field — Claude can suggest regenerating but never does so automatically.
- **§5.1 #3** which explicitly carved out artist pages for this subsystem to author once taste data identifies which artists matter.

### 3.4 Aggregator output JSON shape

`scripts/profile_stats.py` reads `taste/tracks.csv` and prints to stdout:

```json
{
  "row_count": 157,
  "enriched_count": 115,
  "missing_count": 42,
  "top_artists": [
    {"artist": "Disco Lines", "tracks": 12},
    {"artist": "GRiZ", "tracks": 11}
  ],
  "bpm_histogram": {
    "100-110": 0, "110-124": 0, "124-128": 19,
    "128-138": 0, "138-142": 0, "144-150": 0,
    "150-160": 0, "other": 0
  },
  "camelot_distribution": [
    {"key": "10A", "count": 12}
  ],
  "key_split": {"major": 38, "minor": 77, "unknown": 42},
  "feature_means": {
    "energy": 0.83,
    "danceability": 0.55,
    "valence": 0.22,
    "acousticness": 0.06,
    "instrumentalness": 0.18,
    "liveness": 0.27,
    "loudness": -4.1,
    "speechiness": 0.09
  }
}
```

Field semantics:

- `row_count` — total rows in tracks.csv (the dedup count from #6).
- `enriched_count` — count of rows with `source` starting with anything other than `miss:`.
- `missing_count` — `row_count - enriched_count`.
- `top_artists` — full list, sorted desc by tracks, ties broken alphabetically. Not truncated; the consumer (Claude) decides how many to surface in the profile prose.
- `bpm_histogram` — fixed buckets aligned to genre tempo conventions: 100–110, 110–124, 124–128 (tech-house), 128–138, 138–142, 144–150 (dubstep / bass house), 150–160 (drum-and-bass-adjacent), `other` (everything else, including `<100` and `>=160`). Boundaries are left-inclusive, right-exclusive.
- `camelot_distribution` — every Camelot key with at least one row, sorted desc by count. Empty `key_camelot` cells are excluded (counted as `unknown` in `key_split` but not in this list).
- `key_split` — partition of all rows into `major` / `minor` / `unknown` (where `unknown` covers both empty `mode` and `mode` = empty for the GetSongBPM source case).
- `feature_means` — arithmetic mean of each numeric audio feature, computed over rows where `source = "reccobeats"` (the only source that fills these columns per #6 §3.4.1). Rounded to 2 decimals (3 for `loudness`).

### 3.5 `taste/profile.md` shape

Follows master spec §7.4 with a YAML front-matter block prepended:

```markdown
---
generated_at: 2026-04-28T15:30:00Z
row_count: 157
enriched_count: 115
top_artists_at_generation: [Disco Lines, GRiZ, BUNT., jigitz, Zeds Dead]
profile_anchors: [tech-house, melodic-dubstep, dubstep, bass-house]
---

# Destin's taste profile (auto-generated 2026-04-28)

## TL;DR
You listen to 157 tracks across 1 playlist ("EDM"). Your taste centers on …

## BPM clusters
- 124–128 BPM: 19 tracks (tech house — the dominant cluster)
- 144–150 BPM: 28 tracks (dubstep / bass house)
- …

## Key bias
Most common Camelot keys: 10A (B minor) — 12 tracks; 8A (A minor) — 9 tracks; …

## Top artists & subgenres
- **Disco Lines (12 tracks)** — tech-house / house with vocal hooks
- **GRiZ (11 tracks)** — electronic-soul / live-band electronic
- …

## Resources surface area
- Artists with verified Splice "Sounds of" packs: …
- Artists with verified preset packs: …
- Cheapest leverage for buying study materials: …
```

Front-matter fields:

- `generated_at` — ISO 8601 UTC timestamp at write time.
- `row_count` / `enriched_count` — copied from the aggregator JSON; let downstream consumers detect drift cheaply.
- `top_artists_at_generation` — first 5 of the aggregator's `top_artists`, names verbatim. Lets `04-curriculum-and-nudges` notice when the leaderboard shifts.
- `profile_anchors` — array of genre slugs (matching `knowledge/genres/<slug>.md` filenames) that Claude judges to be primary based on the BPM histogram, key bias, and artist mix. v1 expectation: `[tech-house, melodic-dubstep, dubstep, bass-house]` to match the existing genre cheat sheets, but Claude may write a different set if the data warrants.

Body sections are exactly the §7.4 set: TL;DR, BPM clusters, Key bias, Top artists & subgenres, Resources surface area. No additional sections in v1.

### 3.6 Artist page shape

Each `knowledge/artists/<slug>.md` follows the same template as the existing `knowledge/genres/*.md` and `knowledge/theory/*.md` files (front matter → TL;DR → Cheat block → "Read this yourself" → "See also"). Front-matter fields:

```yaml
---
slug: disco-lines
category: artists
tldr: "Vocal-driven tech-house with disco-leaning hooks; one of Destin's heaviest rotations (12 tracks)."
prerequisites: [tech-house]
references:
  - "Splice 'Sounds of Disco Lines' (verify URL at authoring)"
next: griz
---
```

`prerequisites` lists the genre slug(s) the artist primarily inhabits — pointing back into `knowledge/genres/`. `next` points to the next artist page in a default reading order (alphabetical by slug works fine for v1).

Body sections required:

- **TL;DR** — one-paragraph placement: which genre, signature vibe, why they matter to Destin's taste.
- **Cheat block** (bullet list) — typical BPM range; signature sound design (sub-bass character, lead-synth tendencies, drum punch, vocal treatment); most-common Camelot keys derived from the artist's tracks in `tracks.csv`; 2–4 sample track titles from Destin's library so the page grounds in his data.
- **Resources** — Splice "Sounds of [artist]" packs (web-search-verified at authoring); preset packs (Serum, Vital, Phase Plant — verified); notable interviews / breakdowns / YouTube tutorials (verified). Each item linked or marked **"unverified"** or **"no known pack"** if a search returns nothing — never fabricated. See §3.9.
- **Read this yourself** — 1–3 paragraphs of prose connecting the artist to Destin's likely production interests (e.g., "if you're going to write tech-house drops, Disco Lines' vocal-chop hook on 'Honey' is a clean reference for how to layer a chopped vocal over a sidechained bass — see knowledge/genres/tech-house.md").
- **See also** — links to relevant `knowledge/genres/*.md`, other artist pages, and `taste/profile.md`.

### 3.7 Invocation pattern

Conversational, mirroring the established Spotify-pull and audio-enrichment patterns. When Destin says "regenerate the taste profile" / "rebuild profile" / "refresh profile" (or similar), Claude:

1. Runs `python scripts/profile_stats.py` from project root via Bash; captures the JSON.
2. Reads `taste/tracks.csv` directly for any prose-level details the JSON doesn't capture (specific track titles to call out, Camelot key in human-readable form).
3. Writes/overwrites `taste/profile.md` per §3.5.
4. For each of the 14 slugs in §3.2's table, performs a web search to verify or update Splice / preset / breakdown resources, then writes/overwrites `knowledge/artists/<slug>.md` per §3.6.
5. Updates `knowledge/INDEX.md` to list the artist pages under the existing "## artists" section (currently shows "(empty — populated by Session D / Subsystem #7 once the taste profile lands.)").
6. Reports a concise summary: row count, top 5 artists, profile anchors, and the count of artist pages written. Calls out any artist whose resources came back unverified so Destin can investigate manually.

### 3.8 Slug rules

Lowercase, hyphenated, ASCII-only. The §3.2 table fixes the slugs explicitly to remove ambiguity around stylized names (`BUNT.` → `bunt`; `Łaszewo` → `laszewo`; `NGHTMRE` → `nghtmre`). Future artists added in subsequent regenerations follow the same rules.

### 3.9 Resource verification rule

Every external resource claimed on an artist page must be web-search-verified at authoring time. The rule, stated for the spec:

- A Splice pack URL is included only if the search confirms a "Sounds of [artist]" pack is currently listed on Splice.
- A preset pack is included only if a current product page is found (Splice store, the artist's own site, or a reputable third-party seller).
- An interview / breakdown link is included only if the URL resolves and the content is what the page claims it is.
- If a search returns no result, the line says **"no known pack"** or **"unverified — search inconclusive as of YYYY-MM-DD"**, dated. Never invented placeholders.

Resource verification is a fresh search on every regeneration, not a one-time activity. Splice catalogs change.

## 4. What this subsystem must NOT do

- **No modification of `taste/tracks.csv`.** The file is owned by Subsystem #6. This subsystem reads it only.
- **No modification of `taste/playlists.json`.** Owned by the `spotify-services` plugin via Subsystem #5.
- **No new artist pages beyond the §3.2 table for v1.** Long-tail artists do not get pages.
- **No fabricated resources.** Splice packs, preset packs, and breakdown links must be search-verified at authoring time. See §3.9.
- **No automatic regeneration.** Trigger is conversational only (§3.7). Do not add a `/schedule` entry, file watcher, or session-start auto-check that regenerates the profile.
- **No new dependencies.** The aggregator is stdlib only (`csv`, `statistics`, `json`, `sys`).
- **No edits to `knowledge/genres/*.md` or other existing knowledge content** during a profile regeneration. Artist pages link to genre cheat sheets via `prerequisites`, but do not modify them.
- **No deletion of `knowledge/artists/*.md` pages on regeneration.** Existing artist pages remain even if the underlying data shifts; they're rewritten in place but not deleted.

## 6. Implementation notes

- **Histogram boundaries.** The §3.4 buckets (124–128, 144–150) are intentionally aligned to genre tempo conventions documented in `knowledge/genres/tech-house.md` and `knowledge/genres/dubstep.md`. Don't second-guess them by clustering algorithmically — the buckets are pedagogical anchors, not statistically optimal.
- **Missing-value handling.** Every aggregator function that touches audio-feature columns must skip empty cells before averaging. The `feature_means` JSON output is computed over `source = "reccobeats"` rows specifically (per #6 §3.4.1), since GetSongBPM rows leave those columns empty and a mixed average would be misleading. The aggregator's docstring states this contract.
- **Artist name normalization.** When counting tracks by artist, use the `artist` column verbatim (no case-folding, no trimming). The CSV's `artist` is already the primary artist from the Spotify payload (#6 §3.4); collaborators are not split out and the column is treated as canonical.
- **Camelot human-readable in prose.** The aggregator emits Camelot codes (e.g., `10A`); the prose translates to human-readable (`B minor`) using the lookup in `scripts/enrich/camelot.py`. Reuse the existing lookup; do not duplicate it.
- **Web search during artist-page authoring.** Use the WebSearch tool. For each artist, the typical query set is `splice "sounds of" <artist>`, `<artist> preset pack`, `<artist> interview production breakdown`. Two minutes per artist, 14 artists ≈ 30 minutes of resource verification on a regeneration.
- **Order of operations within a regeneration.** Profile.md first (so the aggregator JSON is fresh), then artist pages, then INDEX.md update. If artist-page authoring is interrupted, the profile is still consistent with the JSON it was written from.
- **Reproducibility.** Given the same `tracks.csv`, the aggregator JSON is byte-for-byte identical across runs. The prose layered on top by Claude is not reproducible (LLM nondeterminism), but anchored in the same numbers.
